# 메일링 에이전트 AI 설계를 위한 정보 수집 목록

> 이 문서는 정적 웹 기반 메일링 에이전트 AI 챗봇을 만들기 전에
> 어떤 정보를 미리 정리해야 하는지 체크리스트처럼 모아 둔 것이에요.

## 1. 에이전트 개요
- [ ] 에이전트 이름
- [ ] 대표 사용 상황(어떤 상황에서 이 에이전트를 쓰는지 2~3줄)

## 2. 5요소 정의 (목표·계획·상태·도구·결과)

### 2-1. 목표 (Goal)
- [ ] 한 줄 목표 설명

### 2-2. 계획 (Plan)
- [ ] 한 세션에서 에이전트가 따를 기본 단계
  - 예시: 정보 수집 → 메일 초안 작성 → 사용자 피드백 반영 → 최종 메일 확정

### 2-3. 상태 (State)
- [ ] 코드에서 유지해야 할 상태 항목 목록
  - 예: 사용자 프로필, 수집된 사실들, 메일 초안, 확정 여부 등

### 2-4. 도구 (Tools)
- [ ] LLM 도구 (OpenAI API 사용 방식)
- [ ] 메일 Mock 도구
  - [ ] 도구 이름
  - [ ] 입력값 구조 (수신자, 제목, 본문 등)

### 2-5. 결과 (Done / 종료 조건)
- [ ] "세션이 끝났다"고 보는 기준
- [ ] 최종 산출물 형태 (확정된 메일 본문, 요약, 할일 리스트 등)

## 3. TAO 로그 설계 (Thought · Action · Observation)
- [ ] 1 턴의 정의 (사용자 발화 + 에이전트 응답의 단위)
- [ ] Thought에 남길 내용 (내부 추론 요약 등)
- [ ] Action 종류 (LLM 호출, Mock 메일 도구 호출, 상태 업데이트 등)
- [ ] Observation 내용 (도구 결과, 사용자 피드백 요약 등)
- [ ] TAO를 화면에 어떻게 보여줄지 (별도 패널, 토글 등)

## 4. Mock 메일 도구 설계
- [ ] 도구 이름 (예: MockMailAgent 등)
- [ ] 입력 필드 구조
  - [ ] 수신자(누구에게 보내는지)
  - [ ] 제목
  - [ ] 본문
- [ ] 언제 호출할지 (사용자 표현/버튼 기준)
- [ ] 실제 동작 (어디에 어떻게 로그를 남길지)
- [ ] 연습용 실패 시나리오를 넣을지 여부

## 5. UI/UX 설계
- [ ] 챗봇을 메일링 전용으로 만들지, 일반 질문도 허용할지
- [ ] 우측 패널 설명을 메일링 중심으로 바꿀지 여부
- [ ] 5요소 요약을 어디에 배치할지 (상단 고정, 우측 패널 등)
- [ ] 메일 초안·보낸 기록을 어떻게 보여줄지 (채팅 말풍선 vs 별도 카드)
- 결정: 기존 우측 "사용 방법" 패널 위치를 **TAO 로그 패널(A안)** 으로 사용

## 6. LLM 프롬프트 & 응답 포맷
- [ ] 시스템 프롬프트(에이전트 역할 설명) 문구
- [ ] LLM 응답 형식
  - [ ] 자연어만 줄지
  - [ ] TAO/도구 호출 정보를 포함한 포맷을 쓸지
- 결정: LLM 응답은 최소 3줄 구조를 따른다
  1. `TOOL: askUser` 또는 `TOOL: composeEmail`
  2. `THOUGHT: ...` (에이전트의 한 줄 추론)
  3. 이후 줄: 실제 사용자에게 보여줄 답변 (한국어)


## 7. 저장 범위 & 초기화 정책
- [ ] 새로고침 시 대화·TAO·메일 로그를 모두 지울지 여부
- [ ] localStorage 등에 일부만 남길지 여부 (예: 최근 메일 초안)

## 8. 기타 설정
- [ ] 기본 메일 언어 (한국어만, 혹은 한국어+영어 등)
- [ ] 대표 수신자 시나리오 예시 (담임 선생님, 진로 선생님, 미래의 나 등)
